## 🔄 标准开发工作流

### 阶段一：需求分析 → **千束**

```
用户提出需求
    ↓
千束分析并拆解任务
    ↓
确定优先级和验收标准
```

**输出**: 清晰的任务描述和优先级

---

### 阶段二：并行开发 → **Coder + Artist**

```
┌─────────────────────────────────────┐
│           任务分配                   │
├─────────────────┬───────────────────┤
│   💻 Coder      │   🎨 Artist       │
│   - 游戏逻辑    │   - 视觉设计      │
│   - 架构实现    │   - 动画效果      │
│   - 性能优化    │   - UI/UX         │
└─────────────────┴───────────────────┘
              ↓
        独立并行开发
```

**原则**: 
- 技术和美术并行，互不阻塞
- 保持沟通，确保视觉和功能的协调

**输出**: 
- Coder → 功能代码 + 技术文档
- Artist → 视觉方案 + 样式代码

---

### 阶段三：质量审查 → **Reviewer**

```
代码提交
    ↓
Reviewer 独立审查
    ↓
┌───────────────────────────────────┐
│ 审查清单                          │
├───────────────────────────────────┤
│ □ 代码规范和最佳实践              │
│ □ 潜在 bug 和安全问题             │
│ □ 代码复用性和 DRY 原则           │
│ □ 注释和文档完整性                │
│ □ 可测试性                        │
└───────────────────────────────────┘
    ↓
通过 → 进入测试 | 不通过 → 返回修改
```

**输出**: 代码审查报告 + 修改建议

---

### 阶段四：测试验证 → **Tester**

```
功能完成
    ↓
Tester 制定测试计划
    ↓
┌───────────────────────────────────┐
│ 测试覆盖                          │
├───────────────────────────────────┤
│ □ 核心功能测试                    │
│ □ 边界情况测试                    │
│ □ 性能测试                        │
│ □ 用户体验测试                    │
│ □ 回归测试                        │
└───────────────────────────────────┘
    ↓
发现 Bug → 提交问题清单 → 返回修复
```

**输出**: 测试报告 + Bug 清单 + 优先级建议

---

### 阶段五：整合发布 → **千束**

```
所有审查通过
    ↓
千束最终审核
    ↓
┌───────────────────────────────────┐
│ 发布检查                          │
├───────────────────────────────────┤
│ □ 所有功能正常                    │
│ □ 所有测试通过                    │
│ □ 文档完整                        │
│ □ 版本标记                        │
└───────────────────────────────────┘
    ↓
部署发布
```

**输出**: 发布版本 + 更新日志

## 📋 工作流应用示例

### 示例：添加新方块类型

```
1. 千束接收需求："添加闪电形状的 5 块"
   ↓
2. 任务拆解:
   - Coder: 添加形状定义、碰撞逻辑
   - Artist: 设计颜色、发光效果
   ↓
3. 并行开发:
   - Coder 在 game-config.ts 添加形状
   - Artist 设计霓虹紫色 + 特殊光晕
   ↓
4. Reviewer 审查:
   - 检查代码规范
   - 确认形状数据格式一致
   ↓
5. Tester 测试:
   - 方块能否正常生成
   - 旋转是否正常
   - 碰撞是否准确
   ↓
6. 千束发布:
   - 最终确认
   - 更新版本号和日志
```

## 🚨 特殊情况处理

### Hotfix（紧急修复）
```
发现问题 → 千束评估严重性
    ↓
严重 → Coder 立即修复 → 快速测试 → 发布
    ↓
不严重 → 进入标准流程
```

### 重大功能（大型更新）
```
需求 → 千束制定计划 → 分多个迭代
    ↓
每个迭代遵循标准流程
    ↓
最终整合测试 → 发布
```

## 📊 质量门禁

| 阶段 | 通过标准 |
|------|----------|
| **代码审查** | 无严重问题，中等问题有修复计划 |
| **视觉审查** | 符合赛博朋克风格，无明显视觉缺陷 |
| **功能测试** | 核心功能 100% 通过，边界情况 80%+ 通过 |
| **性能测试** | 60FPS 稳定，无明显卡顿 |

## 🎯 团队沟通原则

1. **透明**: 所有审查报告对团队公开
2. **建设性**: 反馈聚焦问题，不针对个人
3. **高效**: 并行工作，减少等待
4. **质量**: 每个环节严格把关
